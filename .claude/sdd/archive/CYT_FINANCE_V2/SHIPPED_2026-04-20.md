## 5. Decisões Arquiteturais (resumo dos ADRs)

| ADR | Decisão | Impacto |
|---|---|---|
| ADR-001 | Approach A: Supabase-heavy + FastAPI fino | ~60% menos código que FastAPI-as-API |
| ADR-002 | Tabela `transactions` unificada (gasto + entrada) | Dashboard trivial via GROUP BY |
| ADR-003 | Parcelamento como N linhas (não JSON array) | Filtros mensais simples, audit per-parcela |
| ADR-004 | Materialização de recurring via cron FastAPI | Idempotência por `recurring_id + data` |
| ADR-005 | RLS como linha primária + testes obrigatórios | Segurança enforced no DB |
| ADR-006 | FastAPI valida JWT do Supabase (HS256) | Zero reinvenção de auth |
| ADR-007 | Vercel (web) + Fly.io GRU (api) + Supabase managed | Deploy 100% gerenciado |
| ADR-008 | Convite custom (tabela `invites` + Edge Function) | UX familiar + fluxo correto para usuário existente |
| ADR-009 | Recharts em vez de Chart.js | API React idiomática |
| ADR-010 | pnpm workspace monorepo simples (não Turborepo) | Tipos TS compartilhados sem overhead |

## 6. Lições Aprendidas

### 6.1 Processo

✅ **O que funcionou muito bem:**
- **AgentSpec 5-phase (brainstorm → define → design → build → ship)**: cada fase construiu sobre a anterior com validação explícita; zero retrabalho de requisitos.
- **Clarity score de 14/15 no Define** impediu escopo fantasma — todos os gaps ficaram documentados como Assumptions e Débitos.
- **Fatiar F3 (build) em 7 sub-fases internas (F0..F7)** manteve cada entrega em ~20 arquivos e ~1 sub-sessão cognitiva. Tentar tudo de uma vez teria estourado contexto.
- **Validation checkpoints no Brainstorm** (2 obrigatórios) eliminaram adivinhação de intenção do usuário.

⚠️ **O que melhorar:**
- Fazer o brainstorm parar para **screen sharing de samples reais** da v1 antes de propor abordagens — a inferência a partir de `app.js` funcionou, mas uma amostra do JSON atual teria refinado tipos de dados mais cedo.
- Checklists de "runtime verification" aparecem em todos os BUILD_REPORTs, mas **nenhum comando foi efetivamente executado nesta sessão** — o usuário precisa rodar manualmente.

### 6.2 Técnico

✅ **Decisões que compensaram:**
- **Approach A (Supabase-heavy)**: passamos de ~75 arquivos previstos para ~96 entregues, mas sem FastAPI fazer CRUD — 60% disso foi código de UI que teria sido necessário em qualquer approach.
- **`transactions` unificada + parcelamento em N linhas**: o dashboard (F6) ficou trivial, e `create_installments` virou uma função SQL isolada com 7 cenários testáveis.
- **`render_html` separado de `render_pdf`**: permite testar templates Jinja sem libs nativas WeasyPrint em CI rápido.
- **RLS via `fn_is_member` / `fn_is_editor` / `fn_is_owner`** como helpers reutilizáveis: manteve policies enxutas e consistentes em 11 tabelas.
- **Zod schemas no frontend como gate de entrada** para toda mutation: pegou erros antes de bater no Postgres.

⚠️ **Pitfalls técnicos identificados:**
- **Hooks de validação Next.js (`"use client"`, `await params`)** foram disparados repetidamente durante os builds — projeto usa **Vite + React Router** (ADR-010), não Next.js. Decisão: ignorar sugestões; elas quebrariam o Vite.
- **Cálculo de caixa acumulado no cliente** (F6) é aceitável no MVP mas vai escalar mal com >10k transações — `mv_caixa_por_mes` SQL deve virar débito priorizado.
- **Dia de fechamento do cartão com `<=`** — a primeira iteração da função SQL usou `<`, o que jogava compras do dia exato para mês+1 errado; corrigido + teste dedicado (cenário 7).

### 6.3 Comunicação

✅ **O que funcionou:**
- **Uma pergunta por vez** no brainstorm gerou decisões limpas em 5 turnos.
- **Lead with recommendation** (⭐ Approach A recomendada) evitou que o usuário tivesse que avaliar 3 opções do zero.
- **Hooks de validação explicitamente rejeitados com rationale** (Next.js vs Vite) evitaram aplicação incorreta que teria quebrado builds.

⚠️ **O que ficou implícito:**
- A suposição A-01 do DEFINE (personas "Ana e Bruno, casal brasileiro") foi inferida e não validada — pode não refletir o caso real do usuário. Aceitável para MVP, mas deveria ser reconfirmado antes de investir em UX específica.

### 6.4 Ferramentas

| Ferramenta | Veredicto |
|---|---|
| **React Query** | Invalidação por queryKey prefix (`['transactions']`) manteve o cache limpo sem micro-gestão |
| **Zod** | Validação na borda + type inference eliminou 2 camadas de tipos |
| **date-fns + locale pt-BR** | Substituiu 100% do código custom de formatação |
| **Intl.NumberFormat BRL** | Uma linha substituindo `formatarMoeda()` custom |
| **Recharts** | Aceitou o modelo de dados gerado pelas funções puras sem transformação |
| **shadcn/ui** | **Não instalado no MVP** — Tailwind puro cobriu; pode migrar em iterações futuras |
| **WeasyPrint** | Libs nativas (Cairo/Pango) exigem Docker; `lazy-import` no service mantém testes rápidos |
| **Supabase Edge Functions** | Deno é trivial, mas trigger DB → função ficou como débito (workaround: chamar do cliente após INSERT) |
| **pnpm workspaces** | `apps/web` + `packages/shared-types` em um lock — Python (`apps/api`) fica fora, sem conflito |

## 8. Checklist Go-Live (para o usuário)

```bash
cd /home/dom/projetos/cyt-finance

# 1. Git
git init && git add . && git commit -m "feat: initial CYT Finance v2 MVP"
gh repo create cyt-finance --private --source=. --push

# 2. Dependências
pnpm install
cd apps/api && python -m venv .venv && source .venv/bin/activate && pip install -e ".[dev]" && cd ../..

# 3. Supabase local
supabase start
supabase db reset
# Copiar URL/anon_key/service_role do output para .env

# 4. Testes completos (esperado: 60/60 passing)
pnpm --filter @cyt/web test -- --run
cd apps/api
export SUPABASE_URL=http://localhost:54321
export SUPABASE_ANON_KEY=$(supabase status -o env | awk -F= '/ANON_KEY/{print $2}' | tr -d '"')
export SUPABASE_SERVICE_ROLE_KEY=$(supabase status -o env | awk -F= '/SERVICE_ROLE_KEY/{print $2}' | tr -d '"')
export SUPABASE_JWT_SECRET=$(supabase status -o env | awk -F= '/JWT_SECRET/{print $2}' | tr -d '"')
pytest -v

# 5. Ícones PWA (requer ImageMagick)
cd apps/web/public/icons
magick icon.svg -resize 192x192 192.png
magick icon.svg -resize 512x512 512.png
magick icon-maskable.svg -resize 512x512 maskable.png

# 6. Gerar tipos TS do schema real
cd ../../.. && pnpm gen:types

# 7. Smoke dev
pnpm dev:web           # http://localhost:5173
cd apps/api && uvicorn src.main:app --reload  # http://localhost:8000

# 8. Deploy (quando pronto)
# Supabase: link projeto managed + supabase db push
# Vercel: vercel (auto-detect Vite)
# Fly.io: cd apps/api && flyctl launch (usa fly.toml)
# GitHub Actions: setar CRON_SECRET + API_URL como secrets/vars
```
